### 未解决：8K+ 簇 $SDH/$SII/$I30 报错（下轮重点）

真机测试7（含全部已推送修复的 APK 造盘）chkdsk /f 结果：
- **8K/16K/32K/64K**：rc=1，阶段1 完全干净（256 记录 0 坏 0 孤立），但阶段2 报「段5/段B 信息错误 + 文件5 $I30 + 文件9 $SDH/$SII + 文件B $I30 错误」（8 处）
- **512b/2K/4K/auto**：rc=1，**无** $SDH/$SII/$I30 错误，只剩全簇共性的「4 孤立文件 + 插入 C/D/E/F 数据属性」
- **1K**：rc=1，2 处（另类）
- **2K**：rc=3（损坏的主文件表，chkdsk 中止；但纯造盘 JVM 结构干净，疑似 replace/flush 假象）

**JVM 逐字节对比 4K vs 8K 记录 9/11/5**（`NtfsLargeClusterIndexTest`）：$INDEX_ROOT 内容除 cpib(0x0C) 外**完全相同**。8K 唯一结构差异是 root `$INDEX_ALLOCATION` 的 `alloc=8192 real=4096`（index block 4096 < 簇 8192，半簇 INDX），4K 是 `alloc=4096 real=4096`。

**cpib 弯路教训**：`$INDEX_ROOT+0x0C` 曾疑与 boot[0x44] 编码不同（block/512=8 vs 负 log2=-12）。改成 8 后真机 chkdsk 输出与改前逐字节相同 → **cpib 不是根因**，已 revert 回 -12（与 NtfsBoot 用同一 `recordSize()` 解析自洽）。两个子代理给了相反答案（8 vs -12），两次 WebFetch 抓 ntfs-3g 源码被网络墙+页面渲染污染（返回幻觉内容，不可信）。**真因只能靠真实 Windows 8K 簇盘金标准定位**。

**工具封锁**：diskpart/VHD 建 8K 盘触发「找不到虚拟磁盘支持提供程序」（VDS 组件故障，需**重启修复**）；fsutil 需管理员；ntfs-3g 源码网络抓不到（curl 超时/GitHub JS 渲染）。本机 C: 盘是 4K 簇，编码与 App 4K 一致（boot[0x44]=1、idxblk=4096），无法验 8K。

### Bug#4 真相（修正 md 旧判断）

旧 md 说 bug#4 是「`cpr=idxRecSize/clusterSize` 大簇整除为 0」——**方向对，但代码早已改用 ceil**。真正根因经纯 JVM 复现确认有二：
1. `rebuildDirIndex` 有一行 `if (boot.indexRecordSize < clusterSize) return false` —— index block 恒 4096，簇>4096 时该判据**恒真** → 8K+ 簇任何触发多叶的写操作（root 造盘已带 12 系统项，**第一个用户文件插入就触发**）**从第一步就返回 false**。这才是真机「8K 写文件全失败 / $I30/$SDH/$SII 错误」的根因。
2. VCN 语义：INDX 块在 `$INDEX_ALLOCATION` 流内**紧密排布**（第 i 块字节偏移 = `i*4096`），INDX 记录头 VCN 与 subnode 指针的单位是 `if(簇<=块) 块/簇 else 块/512`（ntfs-3g `vcn_size_bits` 规则，经子代理核实权威一致）；属性头 `highest_vcn`/runlist 仍是**簇单位**（两个 VCN 空间不可混淆）。

### 修正 md 对 2k/1k 的误判

旧 md 列「2k MFT 损坏 / 1k 属性记录(0x80) 损坏」为运行时新 bug。**纯 JVM 写路径测试证伪**：512b/1k/2k/4k 全簇写序列（小/大文件、子目录、rename、move、delete、批量 50）+ 深度多叶（写到上限、删一半）**结构全部干净**。那些「损坏」是真机 `replace_full.py` 整卷就地替换的**尾部截断假象**（VeraCrypt 卷末写保护少写 4-8KB），不是代码 bug。

### 关键方法论突破：纯 JVM 复现运行时写路径

旧 md 断言「NTFS 运行时无法在宿主跑（native 加密）」。**证伪**：`VolumeReader` 是 `open` 类、`NtfsFileSystem.mount` 接受任意 reader → 用内存明文卷（ByteArray 后端 + 恒等加密）即可在 JVM 里**确定性**跑完整运行时写路径，用与 chkdsk 同源的 `NtfsBoot`/`NtfsRecordCodec` 解析器逐步验证结构不变量。比真机 chkdsk 快几个数量级、无需 VeraCrypt 挂载、无环境风险。见 `NtfsWritePathTest.kt`。

**已知干净上限**：root 目录用户文件上限约 111（1024B MFT 记录内驻留 `$INDEX_ROOT` 放分隔符的 2 层 B+树约束，3 层保守拒绝），全簇一致——是**优雅失败**（返 false 不写坏盘），非损坏。设备自检的 50 文件远在其下。

## 二、方法论修正（关键教训）

之前几轮（含上个 AI）反复报「VHD 法 9/9 CLEAN」，其实是**假象**：

1. **只读 chkdsk 不做 orphan 扫描**。空 root `$I30` 的系统文件缺失缺陷，只读 chkdsk 完全不报，只有 `chkdsk /f`（触发 orphan 重连阶段）才暴露。
2. **VHD 分区模式 ≠ VeraCrypt superfloppy**。VHD 用 MBR 分区表挂载 NTFS，Windows 走分区路径，宽容；VeraCrypt 把解密数据区当 superfloppy（无分区表，offset 0 = NTFS boot），NTFS.sys 直接严格校验 VBR + 遍历 root 索引。同样字节，VHD /f CLEAN，VeraCrypt /f 报 orphan。
3. **挂载后 dump 不可信**。Windows repair-on-mount 会改写卷（MFT#5 冒 `$TXF_DATA` 0x100 属性即证据）；且 dump 脚本若不做 USA fixup，字段高位混入 USN（如 realSize 读成天文数字，实际低位才是真值）。上个 AI 的「root 索引无法遍历 / realSize 脏值 / 簇大小三故障」全是这类假象。

**唯一可靠验证 = 真实 VeraCrypt 挂载 + chkdsk /f。**

## 三、三个真 bug 详解

### Bug 1：`$SDH` data_length（已 commit）
- 文件：`NtfsSecure.kt` `buildSdhIndexRoot`
- `$Secure`(MFT#9) 的 `$SDH` 视图索引项 data_length（内容偏移 0x02）写成 `0x18`，把尾部 `49 00 49 00`("II" 8 字节对齐填充)误算进 data。正确 `0x14` = SECURITY_DESCRIPTOR_HEADER(hash4+secid4+offset8+len4=20B)。
- 铁证不对称：同记录 `$SII` data_length=0x14 正确 → chkdsk 通过；`$SDH`=0x18 → 报「文件 9 索引 $SDH 检测到错误」。

### Bug 2：boot 0x24 字节序（工作区）
- 文件：`NtfsFormatter.kt` `buildBootRegion`
- `putU32(boot, 0x24, 0x80008000L)` 字节序反了 → 小端展开 `00 80 00 80`：physical drive number(0x24)=0、ext boot signature(0x26)=0。
- Windows 权威值 `80 00 80 00`（0x24=0x80, 0x26=0x80）。改为 `0x00800080L`。
- 分区模式 Windows 用 MBR 忽略此字段（chkdsk 不报）；superfloppy 模式 NTFS.sys 校验 ext boot signature → 缺失 → 卷报「1392 损坏」。

### Bug 3：root $I30 系统文件项 + index block 恒 4096（工作区）
真实 NTFS 的 root 目录 `$I30` 必须含 12 个系统文件索引项（$MFT..$Volume + "."）。App 缺，导致 chkdsk /f 判所有系统文件 orphan。两个子问题：

**3a. 造盘 root $I30 为空** → 恢复 LARGE_INDEX 方案（之前被误判"验证失败"回退）：
- `NtfsRecords.kt`：恢复 `buildIndexRootLarge`（$INDEX_ROOT 占位 END 项）+ `buildRootIndxLeafRecord`（INDX 叶子含 12 项）+ `buildRootDirRecord` 三件套（$INDEX_ROOT + $INDEX_ALLOCATION + $BITMAP）
- `NtfsFormatter.kt`：布局加 rootIndxLcn/rootIndxClusters + 写 12 系统文件项 INDX 叶子

**3b. index block size 恒 4096（关键！）**：
- fsutil 金标准铁证：Windows format 8K 簇 NTFS 的「Bytes Per Index Record Segment」= **4096**，不是 8192。
- 旧 `indexRecordSize = max(4096, bpc)` 让 8K+ 簇 index block=8192/16384/... → superfloppy 下 Windows 无法遍历 root $I30 → 系统文件全 orphan。
- 改：`indexRecordSize` **恒返回 4096**；`indexBufferCode` 支持负数编码（簇>4096 时 boot[0x44]=-12，NtfsBoot.recordSize 解析 2^12=4096）；`rootIndxClusters` 用 ceil 防整除为 0。
- 逐簇 VeraCrypt 就地替换验证：4K(idxBlk=4096) 系统文件 0 orphan；修复前 8K(idxBlk=8192) 系统文件全 14 orphan → 修复后 8K(idxBlk=4096) 系统文件 **0 orphan**。

## 四、Bug 4：运行时大簇 VCN 语义（✅ 已修 2026-07-20 第二阶段）

**JVM 纯内存复现是本阶段关键突破**（见第六节）。此前认为运行时无法在宿主跑，实则
`VolumeReader` 是 open 类、`NtfsFileSystem.mount(reader)` 接受任意 reader——用内存明文卷
（恒等加密 + ByteArray 后端）即可确定性跑完整写路径，用与 chkdsk 同源的解析器逐步验证
结构不变量。`NtfsWritePathTest.kt` 一秒复现，颠覆了上一轮 md 的三处误判。

**复现结论（颠覆 md 假设）**：
- **512b/1k/2k/4k 全部结构干净**：真机报的「2k MFT 损坏 / 1k 属性(0x80)损坏」在纯写路径
  **没有复现**——那是 `replace_full.py` 整卷写尾部截断 / flush 时序造成的假象，不是代码 bug。
- **8k/16k/32k/64k 从第一个文件就全失败**（writeFile 返 false）：这才是唯一真 bug。

**真根因**（比 md 描述的更简单直接）：不是「cpr 整除为 0」，而是
`rebuildDirIndex` 里 `if (indexRecordSize < clusterSize) return false`——index block 恒 4096
（Bug 3b 后），大簇下 4096 < 簇，此判据恒真，把 8K+ 簇的多叶重建**整个拒绝**。而 root
造盘已带 12 系统文件项，任何插入都要触发多叶重建 → 8K+ 簇所有写操作恒失败。

**修复**（`NtfsIndex.kt`，按 ntfs-3g `index.c` 语义，经子代理核实权威一致）：
- 删掉 `if (indexRecordSize < clusterSize) return false` 拒绝分支。
- INDX 块在 $INDEX_ALLOCATION 流内**紧密排布**（第 i 块字节偏移 = `i*indexBlockSize`，恒 4096），
  不再按簇对齐。
- 新增 `vcnPerIndexBlock()`：**两个独立 VCN 空间**——
  - 属性头 highest_vcn / runlist：**恒簇单位**（标准非驻留语义，`clusters-1`）。
  - INDX 记录头 VCN + subnode 指针：`if 簇≤块 → 块/簇；else → 块/512`（大簇=8）。混淆两者
    会让 chkdsk 报 $I30 VCN 越界。
- `clustersForBlocks(numBlocks)` = ceil(numBlocks×块/簇)；`writeIndexAllocationAttr` 加 realSize
  参数（= 块字节总数，可 < 簇对齐 allocSize）。

**验证**：`NtfsWritePathTest` 全簇（512b~64K）通过：写序列（小/大文件/子目录/rename/move/
delete/覆写/批量50）+ 深度多叶（一直写到 root 2 层上限约 111 文件优雅失败，已写入全部可
完整列出、删一半收缩仍完整）。全簇行为同构——证明大簇与小簇走同一条正确路径。
