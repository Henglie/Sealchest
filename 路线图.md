# 密匣 Sealchest · 路线图

> 总目标（恒烈定）：在安卓上**完全复刻 VeraCrypt 的可行功能**，追平并对齐同类项目（EDS / VaultExplorer / CryptoContainer 等）。
> 本文档是能力规划的单一真相源。做完一项回来打勾 + 记落点。
> 守本心见 `上游同步与供应链安全.md`：我们是**严格的 VeraCrypt 容器安卓解析器**，字节级互通是唯一判据。

---

## ▍无 root 边界（先厘清什么做不了）

安卓无 root、纯用户态，以下 VC 功能**物理上做不到**，不在路线图内：
- 整盘 / 分区加密（需块设备访问，无 root 拿不到 `/dev/block`）
- 系统盘加密 / 预引导认证（需改引导链）
- 挂载为系统级块设备（需内核 dm 层 / FUSE root）

我们的等价路径：**文件容器** + **纯用户态解密** + **SAF/内置浏览器暴露文件系统**。VC 的"文件容器"这条线，功能要全复刻。

## ▍阶段拆解（按依赖 + 兼容性硬度排序）

### 阶段 A · 容器兼容性补全（打得开更多容器）
- **A1 keyfile 解锁** — 最硬缺口，用 keyfile 的容器现在直接打不开。
  - 后端：`KeyfileMixer`（纯 Kotlin 复刻 VC `KeyFilesApply`，CRC32 池化模加，字节级一致）✓
  - 接线：`MountManager.unlock` 加 `keyfiles` 参数，openVolume 前混入 ✓
  - UI：解锁界面加 keyfile 入口（`OpenMultipleDocuments` 多选累加去重 → 解锁时当次读字节 → 传 unlock）；换选容器自动清空；有 keyfile 时允许空密码解锁（VC 支持纯 keyfile）✓
  - **待办**：真机回验（带 keyfile 的已知容器能开 + 与桌面 VC 同组 keyfile 结果一致）
- **A2 卷头备份/恢复** — VC 卷尾有备份头组（卷尾 -128KB 区）。主头损坏时用备份头救砖；提供导出/救砖入口。防头损毁致数据永久锁死。
  - 后端：`VolumeHeaderTool`（对容器绝对偏移原始 I/O，不经加解密——卷头是自包含加密块，原样拷贝即可）。布局：主头组 `[0,128KB)`、备份头组 `[size-128KB, size)`。✓
  - 救砖严格序：`verifyOpens` 用密码+keyfile 试开源头（开不了直接抛，绝不用垃圾覆盖）→ `rescueCurrentPrimary` 先把当前主头组导出兜底（可逆）→ 备份头组覆盖主头组。✓
  - UI：解锁区（未挂载态）底部低调「卷头工具」入口 → 导出备份（只读，平时用）+ 从内嵌备份头救砖（需写权限 + 选救援文件保存位 + 错误提示）。✓
  - **待办**：真机救砖回验——故意损坏主头（或用备份头覆盖）后能开卷；导出的备份文件能 `restoreFromFile` 恢复。

### 阶段 B · 容器生命周期（脱离桌面 VC 独立管理）
- **B1 改密码/PIM/PRF/keyfile** — 重写卷头：主密钥不变，用新密码派生新 header key 重新加密卷头区。风险中等（写头区，写坏即锁死 → 必先备份头）。
- **B2 创建新容器** — 选算法/大小/PRF/PIM → 生成随机盐 + 随机主密钥 → 派生 → 加密写卷头 → 数据区随机填充 → 格式化 FAT。最重的"从零写"，但不碰隐藏卷可先做基础版。

### 阶段 C · 隐藏卷（VC 招牌能力）
- **C1 隐藏卷解锁** — 隐藏卷头在卷内固定偏移（标准卷 offset 65536，隐藏卷头紧随主头备份区）。解锁时双套头都试解，密码对哪套开哪套。对上层透明。
- **C2 隐藏卷创建 + 外层保护写** — 创建外层卷 + 内层隐藏卷；写外层时"保护模式"避免踩踏隐藏卷数据区。VC 最精巧的部分，最后做。

### 阶段 D · 文件系统扩展（超出 VC，对齐更多容器实际格式）
- **D1 exFAT 读 → 写** — 大容量容器常用。独立于 FAT，需 exFAT 目录/位图/上转表解析。
- **D2 NTFS 读 → 写** — 最重（MFT / $LogFile / 日志一致性）。写方向风险极高，可能长期只读。

### 阶段 E · 体验对齐同类
- 多容器收藏、快速切换。
- 自动锁定：超时 / 息屏 / 切后台自动 lock，销毁密钥。安全刚需。
- DocumentsProvider 加固、老机型兼容持续打磨。

## ▍验证纪律（每阶段收口必过）

见 `上游同步与供应链安全.md` 的「验证闸门」。凡触碰加解密链路或卷格式的改动，发版前必过：
三 ABI 编译 → 加解密自测三判据 → 已知容器开卷 → **两端互通回验** → keyfile 回验（若涉及）。
任一不过不发版。密匣的信任 = 与桌面 VC 字节级互通，永不为"方便"牺牲兼容。
