# 密匣 Sealchest · 路线图

> 总目标（恒烈定）：在安卓上**完全复刻 VeraCrypt 的可行功能**，追平并对齐同类项目（EDS / VaultExplorer / CryptoContainer 等）。
> 本文档是能力规划的单一真相源。做完一项回来打勾 + 记落点。
> 守本心见 `上游同步与供应链安全.md`：我们是**严格的 VeraCrypt 容器安卓解析器**，字节级互通是唯一判据。

---

## ▍能力分层：默认纯用户态，root 为可选增强（恒烈定）

密匣默认**无 root、纯用户态**跑，这是主线，也是绝大多数用户的路径。但不预先把 root 级能力
排除在路线图外——它们走**「运行时按需申请 root，授予才启用、拒绝则该功能禁用（不影响其余）」**
的可选增强路径。分两层：

**第一层 · 纯用户态（主线，无需任何特权）**
- 文件容器：解锁 / 读写 / 创建 / 改密 / 隐藏卷 / 备份救砖 —— VC「文件容器」这条线全复刻。
- 暴露方式：纯用户态解密 + SAF / 内置浏览器。
- 这一层是产品根基，任何时候都不得因为 root 特性而劣化或复杂化。

**第二层 · root 可选增强（申请到 root 才解锁，属阶段 F，最后做）**
- 整盘 / 物理分区 / 外置 SD 块设备加密（需 `/dev/block` 访问）。
- 挂载为系统级块设备（dm-crypt / FUSE，让任意 app 直接访问，而非仅经 SAF）。
- 系统盘 / 预引导认证：安卓形态与桌面差异极大，即便有 root 也需极谨慎评估可行性，暂记为"待调研"。
- 设计红线：root 能力必须是**纯增量**——检测无 root 或用户拒绝时，仅隐藏/禁用这些入口，
  第一层功能零影响。绝不让主线依赖 root。绝不在未明确告知并取得同意前请求 root。

VC 桌面靠 root/内核做的事，我们能用文件容器等价覆盖的走第一层；只有真正需要块设备的走第二层。

## ▍阶段拆解（按依赖 + 兼容性硬度排序）

### 阶段 A · 容器兼容性补全（打得开更多容器）
- **A1 keyfile 解锁** — 最硬缺口，用 keyfile 的容器现在直接打不开。
  - 后端：`KeyfileMixer`（纯 Kotlin 复刻 VC `KeyFilesApply`，CRC32 池化模加，字节级一致）✓
  - 接线：`MountManager.unlock` 加 `keyfiles` 参数，openVolume 前混入 ✓
  - UI：解锁界面加 keyfile 入口（`OpenMultipleDocuments` 多选累加去重 → 解锁时当次读字节 → 传 unlock）；换选容器自动清空；有 keyfile 时允许空密码解锁（VC 支持纯 keyfile）✓
  - **待办**：真机回验（带 keyfile 的已知容器能开 + 与桌面 VC 同组 keyfile 结果一致）
- **A2 卷头备份/恢复** — VC 卷尾有备份头组（卷尾 -128KB 区）。主头损坏时用备份头救砖；提供导出/救砖入口。防头损毁致数据永久锁死。
  - 后端：`VolumeHeaderTool`（对容器绝对偏移原始 I/O，不经加解密——卷头是自包含加密块，原样拷贝即可）。布局：主头组 `[0,128KB)`、备份头组 `[size-128KB, size)`。✓
  - 救砖严格序：`verifyOpens` 用密码+keyfile 试开源头（开不了直接抛，绝不用垃圾覆盖）→ `rescueCurrentPrimary` 先把当前主头组导出兜底（可逆）→ 备份头组覆盖主头组。✓
  - UI：解锁区（未挂载态）底部低调「卷头工具」入口 → 导出备份（只读，平时用）+ 从内嵌备份头救砖（需写权限 + 选救援文件保存位 + 错误提示）。✓
  - **待办**：真机救砖回验——故意损坏主头（或用备份头覆盖）后能开卷；导出的备份文件能 `restoreFromFile` 恢复。

### 阶段 B · 容器生命周期（脱离桌面 VC 独立管理）
> 顺序（恒烈定）：**先 B2 再 B1**。"能自己建容器"比"改已有容器密码"对日常使用重要得多，也是平替同类的硬门槛；B1 改密码是低频维护操作。B2 打通"从零写卷头"的全套地基后，B1 只是其子集（复用同一套派生+写头逻辑，主密钥不变）。

- **B2 创建新容器（先做）· 已完成代码+编译，待真机验** — 选算法/大小/PRF/PIM → 生成随机盐 + 随机主密钥 → 派生 header key → 加密写主头+备份头两套 → 空 FAT 格式化。最重的"从零写"，不碰隐藏卷的基础版已成。
  - **已落地**：`sc_random.c`（真随机注入，替掉危险归零占位）+ `sc_volume_create_headers`（native 门面，调官方 `CreateVolumeHeaderInMemory` 生成主头 masterKeydata=NULL、备份头复用同一主密钥）+ JNI `nativeSeedRandom`/`nativeCreateHeaders` + `NativeBridge.seedRandom`/`createVolumeHeaders` + Kotlin `VolumeCreator`（预分配→写主/备份头→开卷→写空 FAT）+ `FatFormatter`（空 FAT16/32 镜像）+ 创建向导 UI + 接线。全量 46/46 编译过。
  - **数据区填充基础版决策**：只写 FAT 结构、其余留零（未做全区随机填充）。功能正确、能开卷，但未用扇区是"加密后的全零"，可能泄漏使用量元信息。已留 TODO，真机验通后再决定是否加全区随机填充开关。
  - **待恒烈真机验（唯一判据）**：密匣建容器 → 桌面 VeraCrypt 用同密码/PRF 能原样挂载 + chkdsk 干净 + 能读写。
  - **落点实测（已勘察源码坐实）**：
    - `CreateVolumeHeaderInMemory`（`Volumes.c:1009`）= VC 官方卷头生成，**已编入 CMake**。复用它 → 卷头字节级正确，绝不手写卷头布局（错一 bit 即废卷）。传 `masterKeydata=NULL` 让它自生成随机主密钥（B2），传已有主密钥则用于重加密（B1 复用同一函数）。
    - **头号安全命门 · RNG**：该函数靠 `RandgetBytes` 取随机盐（`Volumes.c:1130`）+ 随机主密钥（`:1068`），而 `sc_stubs.c:82` 的 `RandgetBytes` 是**危险占位**——`memset` 归零假装成功。只读路径永不调它，但 B2 一旦调 → **全零盐 + 全零主密钥 = 灾难性可预测密钥，容器等于没加密**。必须换真随机：Kotlin `SecureRandom` 生成熵 → JNI 注入 native 熵池 → `RandgetBytes` 从池取。这是 B2 第一优先、绝不可妥协项。
    - **FAT 格式化走 Kotlin，不复用 native `Fat.c`/`Format.c`**：二者虽已复制但未编入，且 `FormatFat`/`TCFormatVolume` 依赖设备 I/O 结构与线程/进度回调，移植面大。改用 Kotlin 侧格式化器写空 FAT（复用 T3 已验的 FAT 写能力，完全可控、可测），经 `VolumeReader` 加密写回数据区。
    - 组装：native 新增 `sc_volume_create`（熵注入 + 调官方生成 + 回吐卷头 512B）+ Kotlin `VolumeCreator`（补齐容器文件大小 → 写主头@0/备份头@尾 → 数据区随机填充 → 空 FAT）+ 创建向导 UI。收口判据：密匣建的容器，桌面 VeraCrypt 能原样挂载 + chkdsk 干净。
- **B1 改密码/PIM/PRF/keyfile（后做）** — 重写卷头：主密钥不变，用新密码+keyfile 派生新 header key 重新加密卷头区（主头+备份头都要更新）。风险中等（写头区，写坏即锁死 → A2 备份/救砖已是安全网）。B2 完成后本质是其子集。

### 阶段 C · 隐藏卷（VC 招牌能力）
- **C1 隐藏卷解锁** — 隐藏卷头在卷内固定偏移（标准卷 offset 65536，隐藏卷头紧随主头备份区）。解锁时双套头都试解，密码对哪套开哪套。对上层透明。
- **C2 隐藏卷创建 + 外层保护写** — 创建外层卷 + 内层隐藏卷；写外层时"保护模式"避免踩踏隐藏卷数据区。VC 最精巧的部分，最后做。

### 阶段 D · 文件系统扩展（超出 VC，对齐更多容器实际格式）
- **D1 exFAT 读 → 写** — 大容量容器常用。独立于 FAT，需 exFAT 目录/位图/上转表解析。
- **D2 NTFS 读 → 写** — 最重（MFT / $LogFile / 日志一致性）。写方向风险极高，可能长期只读。

### 阶段 E · 体验对齐同类
- 多容器收藏、快速切换。
- 自动锁定：超时 / 息屏 / 切后台自动 lock，销毁密钥。安全刚需。
- DocumentsProvider 加固、老机型兼容持续打磨。

### 阶段 F · root 可选增强（申请到 root 才解锁，最后做）
> 设计红线见顶部「能力分层」：纯增量，检测无 root / 用户拒绝则仅隐藏这些入口，第一层零影响。绝不在未告知并取得同意前请求 root。

- **F1 root 检测 + 授权门面** — 运行时探测 root（su / Magisk），按需请求；封一层"有 root 才启用"的能力开关。所有 F 阶段功能挂在此门后。
- **F2 块设备 / 分区加密** — 有 root 后可访问 `/dev/block`，对物理分区或整盘做 VC 加密/解锁。安卓形态与桌面差异大，需先真机调研可行性。
- **F3 系统级挂载**（待调研） — 经内核 dm 层 / FUSE 把解密卷挂成系统块设备，让任意 app 无感访问。技术风险高，仅记录方向，不承诺。
- 系统盘 / 预引导认证：安卓引导链与桌面根本不同，暂列"待调研"，不承诺。

## ▍验证纪律（每阶段收口必过）

见 `上游同步与供应链安全.md` 的「验证闸门」。凡触碰加解密链路或卷格式的改动，发版前必过：
三 ABI 编译 → 加解密自测三判据 → 已知容器开卷 → **两端互通回验** → keyfile 回验（若涉及）。
任一不过不发版。密匣的信任 = 与桌面 VC 字节级互通，永不为"方便"牺牲兼容。
