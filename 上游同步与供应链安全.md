# 密匣 Sealchest · 上游同步与供应链安全

> 本文档是密匣对接 VeraCrypt 上游、防旧版供应链攻击的**单一真相源**。
> 每次同步上游、每次改动 `cpp/veracrypt/` 都必须回来更新本文档的「当前锁定版本」与「同步日志」。
> 加密工具的根本信任建立在"用户能审计我们编进去的到底是什么、来自哪里、有没有被篡改"。

---

## ▍守本心（项目定位红线）

密匣是**严格的 VeraCrypt 容器安卓解析器**。这条定位决定一切技术取舍：

- 加解密与卷格式的行为，**必须**与桌面 VeraCrypt 字节级一致。任何"我觉得这样更好"的偏离都是 bug。
- 判据永远是**互通**：密匣写的容器，桌面 VC 能原样打开；桌面 VC 写的容器，密匣能原样读。
- 密码学**实现**只采用上游 VeraCrypt 的 C 源码，我们不自写、不魔改算法。我们只写"把上游算法搬到安卓能跑"的适配层（stub / force-include / JNI 门面）。
- 唯一允许 Kotlin 复刻的是**卷格式规范**层面的定死逻辑（如 keyfile 混入、卷头字段布局）——这些是格式契约、永不随上游变，复刻它们锁的是"格式"不是"实现"，不构成供应链风险。复刻处必须在代码注释里标明对应的上游源文件与算法，供逐字节比对。

关于 **root 可选增强**（路线图阶段 F）的安全红线：
- 默认无 root、纯用户态是主线，是产品根基。root 能力只是**纯增量**叠加，检测无 root 或用户拒绝时仅禁用相关入口，主线功能零影响。绝不让主线依赖 root。
- **绝不在未明确告知用途并取得用户同意前请求 root**。请求时必须说清这一次要用 root 做什么、影响什么。
- root 路径下的任何块设备 / 分区操作是**不可逆、高破坏性**的，必须比文件容器操作更严的确认与备份前置。root 提权本身扩大攻击面，实现时须最小化提权范围、用完即释放，绝不常驻 root 会话。
- 即便走 root，加解密与卷格式仍**必须**是上游 VeraCrypt 的实现 + 字节级互通，本心不变。

扩展格式兼容（exFAT/NTFS…）、安卓端创意，都是守住上述本心之后的后话。

## ▍供应链威胁模型（为什么这份文档必须存在）

加密工具最危险的攻击面不是密码算法本身，而是**"你以为在用的代码"和"实际编进去的代码"之间的缝隙**：

1. **旧版已知漏洞**：锁死在某个含已知 CVE 的旧 VeraCrypt 版本而不自知 → 用户以为安全，实则可被已公开的手段攻破。
2. **上游被投毒**：从不可信镜像、被篡改的 tarball 取源码 → 编进后门。
3. **同步时夹带**：从上游"更新"时，diff 里混入非上游的恶意改动而未被发现。
4. **不可复现构建**：无法证明"这个 APK 就是这份源码编出来的" → 分发环节被替换。

对抗手段（本项目落地）：
- **版本锁定 + 可见**：pinned commit 写进文档 + 编进 APK 关于页，用户随时能核对。
- **来源可验证**：只从官方 GitHub `veracrypt/VeraCrypt` 取源，同步时校验 commit 签名 / release GPG。
- **同步可审计**：每次同步只动 `cpp/veracrypt/`，diff 全部记入「同步日志」，逐条说明。
- **更新有闸门**：同步后必须过「验证闸门」（见下）才允许发版，杜绝"更新即信任"。

## ▍源码分层（哪些跟上游、哪些是我们的）

```
third_party/VeraCrypt/           完整上游 git 快照（比对基准，不编译）
        │  手工挑选 + 记入同步日志
        ▼
app/src/main/cpp/
    veracrypt/                   ← 上游子集（Common/ + Crypto/）。跟随上游，供应链敏感区。
                                    禁止在此手改算法逻辑；只有同步才动它。
    sc_compat.h                  ← 我们的：force-include，补平台类型 + 关 ASM/SSE/SHANI。适配层。
    sc_stubs.c                   ← 我们的：单线程 EncryptionThreadPool stub 等。适配层。
    sc_volume.c / sc_volume.h    ← 我们的：纯净 C 门面，封住 VC 头，只对 JNI 暴露开卷/解密/加密/关卷。
    native_lib.cpp               ← 我们的：JNI 实现。
    shim/                        ← 我们的：缺失头/符号的垫片。
```

判定规则：
- 路径在 `cpp/veracrypt/` 下 = **上游代码**，改动只能来自同步，任何本地改动都是红线（会破坏"字节级一致"且掩盖投毒）。
- 其余 `cpp/*` = **我们的适配层**，可自由改，但不得触碰算法语义。

## ▍同步上游的标准流程（SOP）

> 目标：可控、可审计、可回滚。宁可不跟，不可糊涂地跟。

### 0. 触发条件
- 上游发布含**安全修复**的新版本（最高优先，必跟）。
- 上游新增我们要支持的算法/格式能力。
- 常规版本对齐（低优先，攒着一起做）。

纯功能无关、不涉及我们编入文件的上游改动，**可不跟**——跟得越少，审计面越小。

### 1. 取源并验证来源真实性
```bash
cd third_party/VeraCrypt
git fetch --tags origin
# 校验目标 tag 的签名（VeraCrypt release 由 Mounir Idrassi 的 GPG 签名）
git tag -v VeraCrypt_<新版本>        # 或核对官网公布的 commit hash / SHA256
```
- 必须核对：新 commit hash 与官网 / GitHub release 公布值一致；tag GPG 签名有效。
- 不信任任何第三方镜像、tarball、"整理好的"源码包。

### 2. 圈定影响面（只看我们编入的文件）
```bash
# 列出从 pinned commit 到新 tag，我们实际编入的文件有没有变
git diff --stat 21dba20a..VeraCrypt_<新版本> -- \
  src/Common/Crypto.c src/Common/Volumes.c src/Common/Pkcs5.c \
  src/Common/Xts.c src/Common/GfMul.c src/Common/Endian.c \
  src/Crypto/    # 按 CMakeLists 实际列表逐项
```
- 只有 diff 命中我们编入的文件才需要动 `cpp/veracrypt/`。
- **逐行读 diff**：确认每处改动都能在上游 commit 历史 / PR / changelog 找到出处。无法解释来源的改动 = 疑似投毒，停止同步、上报。

### 3. 复制到编译子集
- 把 step 2 命中的文件，从 `third_party/VeraCrypt/src/...` 复制到对应 `cpp/veracrypt/...`。
- 若上游新增了必须的 .c，加进 CMakeLists 的源文件列表；新增头文件按原相对结构放好（VC 内部用相对 include，结构不能动）。
- **不得**顺手"优化""清理"上游代码。

### 4. 过验证闸门（见下节，全过才算同步成功）

### 5. 更新记录
- 改本文档「当前锁定版本」表 → 新 commit / 版本 / 日期。
- 「同步日志」加一条：日期、旧→新 commit、动了哪些文件、diff 摘要、闸门结果。
- 关于页版本信息随之更新（数据源见下）。
- 升 `versionName` / `versionCode`。

## ▍验证闸门（同步后 / 发版前，必须全过）

任何触碰 `cpp/veracrypt/` 或加解密链路的改动，发版前必须全过以下闸门，缺一不可：

1. **三 ABI 编译通过**：`assembleDebug`，arm64-v8a / armeabi-v7a / x86_64 全绿，无 error。
2. **加解密自测**（App 内「加解密自测」按钮，`VolumeReader.selfTest`）：
   - 往返1 `encrypt(decrypt(C0)) == C0`
   - 加密确实改变数据
   - 往返2 `decrypt(encrypt(X)) == X`
   三判据全过 = encrypt/decrypt 严格互逆。
3. **已知容器开卷**：用**已知密码的 VeraCrypt 测试容器**（多 PRF 覆盖）在真机开卷成功，FAT 引导扇区解出 `MSDOS5.0` OEM 名 + 末尾 `55 AA`。
4. **两端互通回验**：密匣写/改/删 → 桌面 VeraCrypt 打开同容器改动可见 + `chkdsk` 干净。
5. **keyfile 回验**（若容器用 keyfile）：带 keyfile 的已知容器能开；与桌面 VC 用同一组 keyfile 结果一致。

> 闸门 3-5 需真机 + 桌面 VC，属恒烈人工验收。任一不过 → 回滚 pinned commit，不发版。
